# PRD

Живой документ требований к текущей стадии разработки. Обновляется по мере принятия решений.

## Текущий scope

### Формат

Telegram-бот.

### Входные данные (фаза 1)

- Вставленный текст
- URL страницы

### Входные данные (фаза 2 — не в текущем scope)

- Скриншоты / изображения

### Выходные данные

После загрузки и обработки документа пользователь выбирает режим анализа кнопками:

1. **Объяснение простым языком** — полный смысл документа человеческим языком
2. **Важные / рискованные пункты** — положения, важные для безопасности, приватности, платежей, ответственности или неожиданных последствий

Тон бота: **нейтральный**. Бот не рекомендует соглашаться или отказываться.

### UX-модель

Снаружи простой бот; внутри — **document-centric** архитектура (вариант C).

Поток:

1. Пользователь явно создаёт **новый документ**
2. Отправляет текст и/или URL (несколько сообщений могут относиться к одному документу в рамках сессии)
3. Бот сохраняет и обновляет **оригинальный собранный текст**
4. Бот подтверждает готовность документа и предлагает кнопки действий
5. Пользователь выбирает режим анализа
6. Документ остаётся в **истории** для последующего доступа

Правила UX:

- Новые документы создаются **явно** пользователем — без автоматического смешивания разных соглашений
- История документов сохраняется
- Одна платная проверка покрывает один документ и все базовые кнопки анализа внутри него

### Монетизация (текущее направление)

Внешняя модель: пакеты **1, 3 или 10 проверок**.

Внутренняя модель: экономика на уровне документа учитывается отдельно для контроля маржи.

Правила:

- **1 проверка = 1 документ + все базовые кнопки** для этого документа
- **Списание проверки** — при **первом успешном обращении к LLM** в рамках нового документа (обычно этап ingest: преобразование контента в чистый текст; для прямой вставки текста — при первом LLM-вызове, если ingest без LLM)
- На старте нет полноценного бесплатного анализа пользовательских документов
- Бесплатный контент: описание продукта, примеры, demo-кейсы для доверия до оплаты

**Payment providers (roadmap):**

| Этап | Реализация |
|------|------------|
| MVP | Заглушка / ручное начисление проверок (admin) |
| v1 | Cost research на MVP → цены пакетов → ЮKassa (payment link + webhook) |

Дальнейшие способы оплаты не планируем заранее.

**UX покупки:** пакет → способ оплаты → ссылка/invoice (как в референс-ботах: несколько методов, кнопка «Открыть оплату», URL текстом на случай зависания browser).

### Критерии успеха

Проверить, что пользователи:

- находят бота полезным в реальных ситуациях с соглашениями
- готовы платить за проверку документов
- возвращаются за новыми проверками

## Пользовательские сценарии

### Основной — соглашение при регистрации

1. Пользователь видит Terms / Privacy / галочку / popup
2. Открывает бота и создаёт новый документ
3. Вставляет ссылку или текст соглашения
4. Бот подтверждает готовность документа
5. Пользователь нажимает **Объяснить просто** или **Подсветить риски**
6. Читает результат и решает, соглашаться или нет

### Новый документ

1. Пользователь явно вызывает команду или нажимает **Новый документ**
2. Предыдущий документ остаётся в истории; новые загрузки идут в новый документ

### Покупка

1. Пользователь: **Купить проверки**
2. Выбор пакета (1 / 3 / 10)
3. Выбор способа оплаты (на MVP — заглушка или manual)
4. Оплата / ожидание зачисления
5. Создаёт документ; проверка списывается при первом успешном LLM-вызове (ingest)
6. Дальнейшие режимы анализа — без доп. списания

## Требования

### Функциональные

- Явное создание нового документа
- Приём текста и URL как входа
- Сбор нескольких сообщений в один документ в рамках сессии
- Хранение оригинального текста документа
- Кнопки действий после загрузки (простое объяснение, подсветка рисков)
- История документов на пользователя
- Платные пакеты проверок (1 / 3 / 10)
- Абстракция payment providers; MVP — stub/manual, v1 — ЮKassa
- UX: выбор пакета → выбор способа оплаты → payment session (link / invoice)
- Списание одной проверки при первом успешном LLM-вызове в документе (ingest → clean text)
- Повторные режимы анализа в рамках оплаченного документа — без доп. списания

### Нефункциональные

- Стоимость inference должна укладываться в устойчивую unit-экономику
- Ответы понятны людям без юридического образования
- Бот явно указывает, что это не юридическая консультация
- Интерфейс бота поддерживает i18n; язык UI берётся из locale пользователя в Telegram
- Анализ документа выполняется на языке пользователя (тот же locale)
- **Тесты обязательны:** core и business logic покрываются unit/integration-тестами; фича и тесты — в одном этапе/PR

### Язык и локализация

- **UI бота** — интернационализация с первой версии; все системные сообщения через i18n
- **Язык UI и анализа** — из locale пользователя Telegram (`language_code` / доступные языки клиента)
- **Язык исходного документа** — любой; LLM обрабатывает документ и отвечает на языке пользователя

### Модель анализа

**Ingest (при загрузке контента):**

- LLM участвует в преобразовании входа в **чистый текст** (URL, скриншоты; для простой вставки текста — по необходимости)
- При **первом успешном** LLM-вызове в рамках документа списывается **одна проверка**
- Сохраняются оригинал, собранный clean text и метаданные источников

**Analysis (по кнопкам):**

- Результаты (**plain explanation**, **highlights** и т.д.) генерируются **по запросу**
- Сгенерированный результат сохраняется в БД; повтор того же режима — из кэша, без повторного inference
- Дополнительных списаний в рамках уже оплаченного документа нет

## Ограничения

- Inference платный — нет безлимитного free tier на реальные документы пользователей
- MVP должен быть экономически viable (цена с маржой выше измеренной себестоимости документа)
- Продукт должен надёжно работать в главном триггерном моменте (быстрая проверка перед Accept)
- Деплой — только через Docker-образ; сборка образа и тесты — обязательный CI gate на GitHub Actions (локальный Go/Docker не требуются)

## Non-goals

- Юридические советы или рекомендации «соглашаться / не соглашаться»
- Универсальный юридический анализатор как основное позиционирование
- Скриншоты / изображения в фазе 1
- Свободный диалог с документом (follow-up Q&A)
- Выбор тона ответа пользователем (возможно позже)
- Автоматическое определение документа/сессии без контроля пользователя

## Открытые вопросы

- Точные цены пакетов — **v1** (после cost research на MVP)
- Цены пакетов в рублях для ЮKassa
- Нужна ли онлайн-касса / 54-ФЗ — при интеграции ЮKassa
- Максимальная длина документа на одну проверку
- Как показывать demos и примеры до первой покупки
- Стратегия извлечения текста по URL для разных вёрсток страниц
- Конкретная модель LLM по умолчанию (провайдер — через OpenAI-compatible API)
- Fallback locale, если язык пользователя в Telegram не поддерживается
- Поведение при неуспешном первом LLM-вызове (ошибка до списания проверки?)

## Запланированные расширения

- Скриншоты / изображения (фаза 2)
- Уточняющие вопросы по документу («chat with document»)
- Дополнительные режимы анализа
- Выбор тона (нейтральный / осторожный / краткий)
- Другие типы документов при появлении спроса

## Cost research (до финализации цен)

Измерить полную себестоимость сценария на документ — не один вызов модели — для:

- Коротких, средних и длинных документов
- Ingest (первый LLM-вызов → clean text) и всех базовых кнопок анализа
- Ввода текста, URL и (позже) изображений
- Запаса и множителя маржи (целевой диапазон 2–4× от средней себестоимости)
